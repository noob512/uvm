5 Execution Runtime: SIMT‑aware Verification and Cross‑Layer State
5.1 Design

The execution runtime in \sys has three tightly coupled responsibilities: enforcing a SIMT‑aware safety model for device‑side policies, executing those policies efficiently at warp granularity, and coordinating shared state across CPU and GPU through hierarchical maps. Together, these components define what class of GPU‑resident eBPF programs \sys accepts and how they interact with the rest of the system.
SIMT‑aware verification model.
Traditional eBPF semantics assume a scalar, single‑threaded execution context: a single instruction stream runs on a CPU core, and the verifier reasons about that scalar control flow and its memory accesses. GPU kernels, in contrast, execute under the Single Instruction Multiple Threads (SIMT) model, where groups of threads called warps execute in lockstep and diverging control flow between lanes leads directly to performance loss and, in the worst case, deadlock when combined with synchronization primitives.

To reconcile these models, \sys defines a restricted class of GPU‑side eBPF programs with explicit SIMT‑aware semantics. At the core of this model is a simple type discipline that classifies values as warp‑uniform or lane‑varying. Warp‑uniform values are guaranteed to take the same value across all lanes in a warp; in practice these are fields of the hook context that derive from driver or kernel state, such as region identifiers, kernel identifiers, tenant identifiers, or static configuration parameters. Lane‑varying values may differ across lanes; they typically arise from per‑thread execution state, such as effective memory addresses, per‑thread indices, or lane identifiers.
The verifier performs a conservative dataflow analysis over the policy program to track the uniformity of each value. Context fields are annotated as uniform or varying via BTF metadata. Arithmetic and logical operations propagate these annotations: combining two uniform values yields a uniform result, any operation involving a varying operand yields a varying result, and helper or kfunc calls have their return uniformity declared in their specifications. This analysis is intentionally conservative: any helper whose uniformity cannot be statically guaranteed is treated as returning a varying value.

Using this classification, \sys imposes strict control‑flow constraints. All conditional branches and loop bounds in GPU‑resident policy programs must depend only on warp‑uniform values. Any attempt to branch or loop on a lane‑varying value causes the program to be rejected at load time. This ensures that, for any accepted program, all lanes in a warp follow the same control‑flow path through the GPU‑side policy code. Lane‑varying values are still allowed, but only as inputs to arithmetic, aggregation, or summary computations; they cannot directly steer control flow.
The verifier further constrains side effects on shared state. Writes to shared eBPF maps and emission of hints or scheduling decisions form the observable effects of a policy program. \sys requires that such effects be expressible as warp‑level operations: updates must be keyed by warp‑uniform keys, and their values must be computed from warp‑uniform quantities or warp‑level aggregates of per‑lane contributions. Direct per‑lane updates to maps keyed by lane‑varying values or direct per‑lane emission of hints are not permitted in GPU‑BPF programs and are rejected by the verifier. This restriction defines a clear semantic boundary: GPU‑side policies observe and act at warp granularity, but can incorporate per‑thread information only through well‑defined reduction operators.

Finally, the SIMT‑aware verifier enforces a extend resource bounds model and forbids dangerous synchronization patterns. Each policy type (for example, memory hook, scheduling hook, device‑side instrumentation hook) has a per‑program budget not only on the number of eBPF instructions, but also on map operations, cetain memory access helper calls, and memory ops per instruction to ensure it does not take over the memory bandwidth of the GPU, chosen conservatively based on the trigger frequency of each hook and the latency budget of the underlying path. \sys also bans GPU‑wide barriers and device‑level synchronization constructs that could violate forward progress if used incorrectly, as well as non‑uniform atomic patterns that could introduce inter‑warp dependency cycles. The result is a closed, statically defined class of GPU‑side eBPF programs whose warp‑uniform control flow and bounded resource consumption are guaranteed by our analysis.

This design deliberately trades per‑lane control and state for analyzable warp‑level behavior; in our experience, the memory placement and scheduling policies we target only require warp‑level decisions and statistics, making this tradeoff acceptable in practice. 
Warp‑aggregated execution model.
Given this SIMT‑aware semantics, \sys defines a warp‑aggregated execution model for device‑side policy handlers. Rather than executing a scalar handler independently on each thread in a warp, \sys treats each GPU‑BPF handler as a warp‑level function that executes once per warp at each attach point. This design directly exploits the warp‑uniform control‑flow guarantee enforced by the verifier and eliminates redundant work.

Conceptually, \sys divides each device‑side hook invocation into a lane‑local phase and a warp‑policy phase. In the lane‑local phase, each lane in a warp may perform arbitrary scalar computation over its lane‑varying inputs, such as effective addresses or per‑thread indices. These computations must not influence control flow or directly update shared state; they serve only to produce per‑lane contributions (for example, access counters, local feature hashes, or partial metrics). The results of the lane‑local phase are then aggregated at warp scope using GPU warp primitives, such as ballots and shuffles, to form warp‑level summaries.
In the warp‑policy phase, a designated warp leader (often lane 0 or the first active lane) executes the GPU‑side eBPF handler exactly once, using as inputs the warp‑uniform context together with the aggregated warp‑level summaries. The handler computes a warp‑level decision—such as updating a per‑region hotness counter, emitting a prefetch hint, or recording warp‑level progress—and issues shared‑state updates via approved helper functions. Any per‑lane effects visible to the application (for example, whether a particular warp should skip or delay certain work units) are derived from this warp‑level decision and broadcast back to all lanes using warp‑wide communication primitives.

This two‑phase model allows rich per‑thread computation to inform policy decisions, while guaranteeing that all observable side effects and control‑flow decisions are warp‑scoped, warp‑uniform, and bounded. Under the constraints enforced by the SIMT‑aware verifier (uniform control flow, warp‑level side effects, bounded resources), warp‑aggregated execution yields the same warp‑level observable behavior as a design that would run the scalar handler on each lane and aggregate their effects. In practice, \sys adopts the warp‑aggregated model directly as the official semantics of GPU‑resident eBPF: programmers reason in terms of warp‑level policies and summarization, rather than per‑thread handlers.
Hierarchical cross‑layer map abstraction. The third key component of the design is a hierarchical map abstraction that gives policies a unified logical view of cross‑layer state while allowing the runtime to optimize placement and synchronization across CPU and GPU memory hierarchies. Policy authors interact with eBPF maps through the standard map interfaces: maps are defined in the policy in terms of key and value types, and used to store state such as per‑region access statistics, per‑tenant scheduling metadata, configuration parameters, or instrumentation counters. Logically, each map in \sys is a single key‑value store that is accessible from host‑side driver hooks, device‑side handlers, and the user‑space control plane. Physically, however, the runtime is free to realize a map as a hierarchy of shards distributed across host DRAM, GPU global memory, and on‑chip memories such as shared memory or SM‑local storage. The primary design goal is to hide the complexity of this hierarchy from policy authors while still enabling efficient, low‑latency access from hot paths.
Consistency Model and Semantics.

\sys maps are designed explicitly as soft state, following eBPF's philosophy of per-key atomicity and weak consistency. Each individual map lookup or update operation is atomic with respect to a given key, meaning policies will never observe partially updated values. However, there are no global ordering guarantees across concurrent updates by GPU kernels or host-side drivers; if multiple contexts concurrently update the same key, the eventual result is determined by the most recent writer.

Rather than enforcing strong consistency or transactional semantics, \sys provides a simple snapshot-based consistency model aligned with policy requirements. Device-side handlers can perform frequent updates locally (e.g., incrementing warp-level counters, updating hotness statistics), and the runtime periodically merges these local updates into canonical map instances located in GPU global memory or host DRAM. Host-side handlers and user-space control-plane logic subsequently observe coherent snapshots of map contents at these well-defined merge points—typically at GPU kernel completion boundaries or explicitly triggered synchronization events.

This relaxed consistency model is adequate for policy decisions that rely on aggregated statistics, approximate tenant accounting, and similar soft-state metadata. Stale data can potentially degrade decision quality (e.g., suboptimal memory eviction or scheduling), but cannot break critical correctness invariants like virtual address mappings or GPU memory integrity, which remain enforced by the underlying GPU driver and MMU mechanisms.

5.2 Implementation

The implementation of \sys’s execution runtime realizes the above design by extending the existing kernel eBPF verifier, adding a device‑side SIMT‑aware verification and JIT pipeline, and building a hierarchical map runtime that manages state placement and synchronization.

Host‑side verification and integration.
On the host, \sys reuses the unmodified Linux kernel eBPF verifier for all driver‑resident policy programs. Host‑side GPU policy types (for example, memory and scheduling hooks) are declared using BTF with dedicated program types and kfunc descriptors. The standard verifier enforces memory safety, type safety, bounded loops, and correct helper usage as it does for existing eBPF subsystems. \sys does not change host‑side verification rules; it relies on the existing verifier as part of the TCB.

Verified host‑side programs are then attached to hook tables exported by \sysdriver. These hook tables expose struct‑ops style function pointer slots for each policy decision point (for example, region admission, region access, region removal, kernel submit, kernel admit). Attaching a policy consists of updating these slots to point to trampoline stubs generated by the kernel JIT for the corresponding eBPF programs. Host‑side policies invoke standard and \sys‑specific kfuncs to access GPU driver state and interact with kernel‑resident maps.

Device‑side SIMT‑aware verifier.
For device‑side programs, \sys implements a SIMT‑aware verifier as a kernel module that reuses the existing verifier’s infrastructure for control‑flow graph construction and basic safety checking but extends it with GPU‑specific analyses and constraints.

The verifier first classifies all context fields as warp‑uniform or lane‑varying based on hook definitions. It then performs a forward dataflow analysis over the eBPF intermediate representation, propagating uniformity information through arithmetic and logic operations and incorporating helper specifications that indicate whether a helper’s return value is uniform or varying. To keep the analysis tractable and conservative, helpers that cannot be proven to return uniform values are treated as returning varying values.

Once uniformity annotations are computed, the verifier scans all conditional branches and loop bounds to ensure they depend only on uniform values or on the results of approved warp‑level aggregation helpers. Any branch or loop whose condition depends on a lane‑varying value is rejected. Similarly, the verifier inspects all map operations and helper calls that have observable effects, ensuring that their keys are uniform and their values derive from uniform or aggregated quantities. Direct per‑lane map updates keyed by lane‑varying values are disallowed. While this restriction rules out certain per‑thread logging patterns, it greatly simplifies both verification and the warp‑aggregated execution model, and in practice suffices for the policies we study.

The verifier also enforces per‑program resource budgets. Each device‑side program type is assigned an instruction budget, a maximum number of map operations and helper calls, and an approximate bound on memory accesses per invocation. These budgets are enforced by counting instructions and calls along all verifier‑reachable paths in the control‑flow graph, leveraging the standard eBPF bounded‑loop analysis. Finally, the verifier rejects programs that invoke forbidden synchronization primitives or attempt to access device control registers or page tables via kfuncs.

Because there is no runtime sandbox on the device, any program that fails verification is simply not loaded; programs that pass verification are trusted to execute directly on the GPU under the hardware’s native protection domains.

Warp‑aggregated JIT and trampolines.
Device‑side execution relies on a JIT backend that translates verified eBPF bytecode into GPU‑specific intermediate code (for example, PTX) and injects it into target kernels via trampolines. The JIT backend leverages the uniformity annotations and the division between lane‑local and warp‑policy phases to generate efficient code.

At each device‑side attach point (for example, kernel entry, a selected global memory operation, or a warp‑wide synchronization point), \sys’s instrumentation pass inserts a trampoline. The trampoline performs three tasks. First, it gathers the hook context, including both warp‑uniform fields and lane‑varying inputs such as effective addresses or local indices. Second, it executes the lane‑local phase: per‑lane computations over local inputs that produce compact per‑lane contributions. Third, it aggregates these per‑lane contributions into warp‑level summaries using warp intrinsics such as ballots, reductions, and shuffles, and invokes the warp‑policy phase by executing the compiled GPU‑BPF handler on a designated warp leader.

The GPU‑BPF handler itself is compiled as a scalar function that assumes its inputs are warp‑uniform values and pre‑aggregated summaries. The JIT aggressively inlines eBPF helper calls and small helper functions, specializes generic helper logic for known map types and access patterns, and hoists invariant computations out of hot loops. Because the SIMT‑aware verifier has already guaranteed bounded loops and uniform control flow, the backend can avoid inserting complex runtime checks and focus on register allocation and instruction scheduling to minimize register pressure and occupancy impact.

To further control overhead in very hot paths, the runtime optionally supports sampling at the trampoline level: for selected attach points, only a fraction of warps invoke the full handler, with the rest executing a fast path that either skips instrumentation or performs a lightweight approximate update. Sampling decisions and rates are tunable via configuration maps and are not part of the core semantics.

Hierarchical map runtime and synchronization.
The hierarchical map abstraction is implemented as a runtime layer that interposes on all eBPF map operations issued by \sys policies. Each logical map is described by a metadata structure that records simple usage hints (e.g., configuration vs. statistics) and whether the map is primarily read or written from host or device. These hints are not part of the semantics and incorrect hints only affect performance, not correctness.

Based on these hints, \sys's map manager decides how to realize the logical map across physical memory tiers. For low‑frequency configuration maps, the primary copy is resident in host DRAM as a standard kernel eBPF map. Device‑side reads see a cached snapshot, updated at well‑defined synchronization points, such as kernel launch or explicit refresh barriers triggered by the control plane. For high‑frequency statistics maps, the primary copy is allocated in GPU global memory, possibly with per‑SM or per‑warp shards stored in on‑chip memories; host‑side views are maintained as aggregated snapshots and refreshed periodically or on demand via bulk transfers.
