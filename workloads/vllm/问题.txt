在进行实验时发现如下问题
1.首先我们在每次重启虚拟机后都执行如下命令
`# 1. 停止持久化服务（如果它没启动会报错，正常现象，不用管）
sudo systemctl stop nvidia-persistenced

# 2. 依次卸载现存驱动（一旦报错，请立刻停止并检查输出）
sudo rmmod nvidia_uvm
sudo rmmod nvidia_drm
sudo rmmod nvidia_modeset
sudo rmmod nvidia_peermem
sudo rmmod nvidia

# 3. 终极验证：必须没有任何输出。如果有输出，说明有进程死锁了 GPU。
lsmod | grep nvidia`

`cd ~/nvidia-uvm-gpu/kernel-module/nvidia-module

# 1. 插入底层核心驱动
sudo insmod kernel-open/nvidia.ko

# 2. 插入 UVM 驱动（包含 eBPF 钩子的核心）
sudo insmod kernel-open/nvidia-uvm.ko

# 3. 验证钩子是否在内核成功注册（预期看到 "UVM: bpf_struct_ops initialized"）
sudo dmesg | tail -n 15 | grep UVM`
这会替换当前的nvidia内核驱动到575版本

2.如果每次重启虚拟机后第一次运行命令为
uv run python configs/serve_bench.py --mode uvm --prompts 100 -o results/exp3/uvm_base.json
即只使用uvm模式但不挂在ebpf程序，则此时的数据结果会很糟糕，数据为/home/ubuntu/nvidia-uvm-gpu/workloads/vllm/results/exp3/uvm_baseline.json
此时如果再挂载ebpf程序，数据仍然无法获得改善，数据为/home/ubuntu/nvidia-uvm-gpu/workloads/vllm/results/exp3/uvm_ebpf_base.json

3.但是如果重启虚拟机后第一次运行程序时已经先挂载了ebpf程序，此时运行下列命令
uv run python configs/serve_bench.py --mode uvm --prompts 100 -o results/exp3/uvm_ebpf.json
此时会发现数据会得到大幅度改善，数据结果为/home/ubuntu/nvidia-uvm-gpu/workloads/vllm/results/exp3/uvm_ebpf_base_3.json
之后即使卸载ebpf程序后再运行uv run python configs/serve_bench.py --mode uvm --prompts 100 -o results/exp3/uvm_ebpf.json
即运行无ebpf程序的纯uvm模式，数据仍然保持有较大的改善，数据为/home/ubuntu/nvidia-uvm-gpu/workloads/vllm/results/exp3/uvm_base_2.json

4.首先先检查/home/ubuntu/nvidia-uvm-gpu/workloads/vllm/configs/serve_bench.py，检查一下是否是记录数据的程序存在问题
其次可以结合docs中的文档，详细解释一下上述结果产生的原因

5.由于现在bpftool存在问题，无法使用，因此要在extension/prefetch_adaptive_sequential程序和extension/attention_aware_eviction中添加一些代码来在运行时判断ebpf程序是否真的被完整卸载了

6.由于上述命令运行都耗时较长，所以不要在查找问题时运行上述命令，上文描述与数据均属实
